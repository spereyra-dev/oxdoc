# OOXML Model

OOXML files such as `.docx`, `.xlsx`, and `.pptx` are ZIP containers with XML parts and relationship files.

`oxdoc` does not parse the whole package. It targets the parts needed for extraction.

## Common Package Parts

| Path | Purpose |
| --- | --- |
| `_rels/.rels` | Root relationships, including the office document relationship. |
| `docProps/core.xml` | Core metadata such as author and creation time. |
| `docProps/app.xml` | Application metadata and document statistics. |

## DOCX Parts

| Path | Purpose |
| --- | --- |
| `word/document.xml` | Main document body. |
| `word/_rels/document.xml.rels` | Relationships from the main document part. |
| `word/vbaProject.bin` | Macro project part when present. |

## XLSX Parts

| Path | Purpose |
| --- | --- |
| `xl/workbook.xml` | Workbook and sheet list. |
| `xl/_rels/workbook.xml.rels` | Sheet relationship mapping. |
| `xl/sharedStrings.xml` | Shared string table. |
| `xl/worksheets/sheetN.xml` | Worksheet data. |
| `xl/vbaProject.bin` | Macro project part when present. |

## Why Relationships Matter

Office parts usually follow conventional paths, but robust OOXML tools should read relationships instead of assuming all paths. `oxdoc` follows the root office document relationship and workbook sheet relationships in the MVP.

Relationship targets are resolved as package-internal OOXML part paths. External URLs, URI schemes, backslashes, NUL bytes, and `..` segments that escape the package root are hard errors.

## Tolerant Input

Real documents are often generated by different Office suites and third-party exporters. The parser should:

- Extract useful data before malformed input.
- Return warnings for recoverable issues.
- Use hard errors when required ZIP/XML parts cannot be read.
